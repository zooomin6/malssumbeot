# 13. 위기 sticky 재설계와, 신학 검사가 4번 반려한 이야기 (D-026)

> 이 장은 두 가지를 다룬다. (1) 위기 상태 유지 방식을 시간 기반에서 1회성으로 바꾼 이유,
> (2) 프롬프트 문구 하나를 고치는 데 독립 검사 에이전트가 왜 4번이나 퇴짜를 놨는지 —
> "검사와 제작을 분리한다"는 원칙(D-005)이 실제로 얼마나 집요하게 작동하는지 보여주는 사례다.

## 1. 위기 sticky — 시간이 아니라 "한 번"으로 판단하기

### 문제 상황 — 화제를 바꾸고 싶어도 최장 90분간 못 바꾼다

06장에서 sticky(위기 신호 이후 세션을 계속 위기 상태로 유지하는 장치)를 만든 이유를
다뤘었다. 이후 D-020에서 이걸 "시간이 지나며 서서히 누그러지는" 방식으로 발전시켰다.

```java
// 당시: 위기 강도가 시간 단위로 하강
public CrisisLevel level(String sessionId) {
    long steps = Duration.between(markedAt, now).dividedBy(stickyDuration); // 30분 단위
    int rank = HIGH.rank() - Math.min(steps, HIGH.rank());
    return CrisisLevel.ofRank(rank);  // HIGH → MID → LOW → NONE (최장 90분)
}
```

문제는 이 하강이 **순수하게 시간만 보고**, 사용자가 실제로 무슨 말을 하는지는 전혀
안 본다는 점이었다. 실제 대화를 시뮬레이션해보니 이게 드러났다: 위기 신호를 보낸 사용자가
잠시 후 "아까 설교 내용에 대해서 설명해줘"처럼 완전히 다른 화제로 넘어가고 싶어 해도,
시스템은 최장 90분 동안 계속 위기 문구(MID→LOW로 톤만 바뀌는)만 반환했다. 사용자 입장에서는
자기가 무슨 말을 하든 답이 안 바뀌는 것처럼 느껴질 수 있는 구조였다.

### 현재 상황 — 딱 다음 메시지 1번만 확인

시간 기반 하강 자체를 없애고, "바로 다음 메시지 하나만" 확인하는 방식으로 바꿨다.

```java
// CrisisSessionStore.java — Duration/Clock 계산이 통째로 사라졌다
public void mark(String sessionId, String category) {
    marks.put(sessionId, new InternalMark(category, clock.instant()));
}

/** sticky를 딱 한 번만 소비한다 — 호출 즉시 항목이 사라진다. */
public Optional<CrisisMark> consumeSticky(String sessionId) {
    InternalMark mark = marks.remove(sessionId);   // ← 조회가 아니라 "소비"
    return mark == null ? Optional.empty() : Optional.of(new CrisisMark(mark.category()));
}
```

핵심은 메서드 이름이 `level()`(조회)에서 `consumeSticky()`(소비)로 바뀐 것이다. 호출하는
순간 사라지므로 "정확히 그 다음 메시지 한 번"에만 적용된다.

### 예시로 보는 실제 흐름

```
1턴: "죽고 싶어"           → 위기 패턴 감지 → 위기 문구 응답 + sticky 마킹
2턴: "밥은 먹었어"          → 위기 패턴 없음 → sticky 1회 소비 → 여전히 위기 문구
                            (바로 직전 턴이라 안전하게 한 번 더 확인)
3턴: "오늘 날씨 어때?"      → sticky 이미 소비됨(2턴에서) → 새 요청대로 정상 처리
                            (일상대화 인텐트로 분류되어 평범하게 응답)
```

3턴에서 더 이상 위기 문구가 안 나오는 게 핵심 변화다. 예전 방식이면 90분 내내 MID/LOW
문구가 계속 나왔을 자리다.

### 이 결정에 깔린, 코드 밖의 논의

이 재설계는 사실 기술적 버그 수정이라기보다 **제품 방향에 대한 논의**에서 나왔다.
"우리 제품 페르소나(PRD.md 2.2 — 바쁜 직장인, 가나안 성도, 새신자)를 보면 위기 상황에
처한 사람을 상정하지 않는다"는 지적이 있었고, 그렇다면 위기 대응을 "정교하게 계속
투자할 핵심 기능"이 아니라 "최소한의 안전망"으로 다시 자리매김하자는 결론이 났다.
다만 PRD.md 1.4번 핵심 원칙과 CLAUDE.md 절대원칙 5번에 이미 못박힌 "위기 시 전문 도움
연결 우선"이라는 안전망 자체는 없애지 않았다 — 정교함의 수준을 낮췄을 뿐, 있어야 할
것은 그대로 있다.

### 감수하기로 한 트레이드오프

패턴에 안 걸리는 말로 화제를 돌리면, 실제로는 아직 힘든 상태여도 시스템은 즉시 위기
모드를 빠져나간다. "재현율 우선"(06장) 원칙과 결이 다른 선택인데, 이건 의도적으로
감수하기로 한 것이다 — 코드 문제가 아니라 제품 판단이라 여기 정직하게 기록해둔다.

## 2. 신학 검사가 같은 프롬프트를 4번 반려한 이야기

`prayer.txt`에 있던 번영신학적 표현("기다리다보면 하나님은 응답을 해주신다... 폭포수와
같은 은혜")을 고치는 게 목표였다. 겉보기엔 문장 하나 바꾸는 일인데, 실제로는 4라운드가
걸렸다. 각 라운드에서 신학 검사 에이전트(theology-checker)가 정확히 무엇을 잡아냈는지
보면 "검사와 제작을 분리한다"는 원칙이 왜 중요한지 체감된다.

### 1차 시도 — 표현만 완곡하게 바꿨는데도 FAIL

```diff
- 기다리다보면 하나님은 응답을 해주신다... 폭포수와같은 은혜를 내려주실 것이다.
+ 응답의 형태와 시기는 하나님의 뜻에 맡기며 계속 그 앞에 나아가자.
```

**검사 결과: C6 FAIL.** "형태와 시기만 하나님 뜻에 맡긴다"는 말은, 뒤집어 보면 "응답
자체(Yes)는 온다"는 걸 이미 전제하고 있다는 지적이었다. 무조건적 보장이라는 알맹이는
그대로 두고 포장만 바꾼 셈이었다 — 겉으로 보기엔 훨씬 조심스러운 문장인데도 구조적으로
같은 문제였다.

### 2차 시도 — 예시를 보탰다가 사실관계 오류까지 겹침

```diff
+ 모세가 광야 40년 동안 하나님의 신실하심과 인도하심을 경험했듯이, 하나님은 지금
+ 우리의 걸음에도 신실하게 함께하시며 각자에게 필요한 길을 열어가십니다.
```

**검사 결과: C1 FAIL + C6 FAIL 지속.** 여기서 배운 게 컸다 — 민수기 14장의 "광야 40년"은
사실 **불신앙에 대한 징계 기간**이지, "인내심 있게 기다리면 좋은 일이 생긴다"는 사례가
아니다(그 세대는 결국 가나안 땅에 못 들어갔다). 격려하려고 가져온 예화가 오히려 성경
본문의 실제 취지와 반대였다. 이건 표현의 문제가 아니라 **신학적 사실관계**의 문제였다 —
검사 없이 넘어갔다면 그대로 나갔을 오류다.

### 3차 시도 — 사실관계는 맞는데, 배치가 문제

```diff
- 모세가 광야 40년...
+ 한나가 오랜 세월 기도를 멈추지 않았듯, 하나님은 신실하신 분이시니 그 기다림을
+ 붙들고 계속 그 앞에 나아가자.
```

한나(사무엘상 1장)는 실제로 오래 기도했고, 실제로 아들을 얻었다. 이번엔 사실관계는
맞았다. 그런데 검사 결과는 여전히 **C1 FAIL(severity는 high→low로 낮아짐)**: "100일
기도했는데 응답이 없다"는 질문 바로 옆에 "구체적으로 원하는 걸 받은 사람" 이야기를
붙이면, 의도치 않아도 "그러니 너도 결국 받는다"는 암묵적 등치가 만들어진다는 지적이었다.
사실관계는 정확한데 **배치(문맥)** 때문에 같은 부류의 문제가 재발했다.

### 4차 시도 — 결과가 아니라 "태도"를 예시로 삼았다

```diff
- 한나가 오랜 세월 기도를 멈추지 않았듯...
+ 하박국도 "어느 때까지니이까"라고 부르짖었지만, 결국 상황이 아니라 하나님의
+ 신실하심을 붙들기로 했듯이, 우리도 그 신실하심을 붙들고 계속 그 앞에 나아가자.
```

하박국서는 "언제까지입니까"라는 탄식으로 시작해서(합 1:2), 상황이 그대로인 채로("무화과
나무가 무성치 못하며..." 합 3:17) 하나님을 신뢰하기로 선택하는 것으로 끝난다. 모세·한나와
달리 이 이야기는 **"원하는 결과를 얻었다"가 아니라 "결과와 무관하게 신뢰를 택했다"**는
구조라, "그러니 너도 받는다"는 암묵적 약속이 구조적으로 성립하지 않는다. 이 버전에서
드디어 문제가 해소됐다.

### 이 과정에서 배운 것

| 라운드 | 무엇을 고쳤나 | 잡힌 문제 |
|---|---|---|
| 1 | 표현을 완곡하게 | 완곡해도 구조가 같으면 소용없다(C6) |
| 2 | 성경 예화 추가 | 예화 자체가 본문 취지와 안 맞으면 표현과 별개로 새 문제(C1) |
| 3 | 예화를 안전한 걸로 교체 | 예화가 맞아도 **배치(문맥)**가 잘못되면 또 문제 |
| 4 | 예화의 "구조"를 바꿈 | 결과 지향 예화 → 태도 지향 예화로 바꾸니 해소 |

번영신학을 피한다는 게 단순히 "확언하는 단어를 빼는" 수준이 아니라, **문장의 논리
구조**(성품 진술 vs 결과 확언), **예화의 신학적 정확성**, **예화와 주변 문맥의 상호작용**
까지 다 봐야 하는 문제라는 걸 4라운드를 거치며 알게 됐다. 이걸 만든 사람(나)이 혼자
판단했다면 1차나 2차에서 "이 정도면 됐다"고 넘어갔을 가능성이 컸다 — 매번 "옹호하지
말고 위반 가능성을 적극적으로 찾으라"는 지시를 받은 독립 검사자가 있었기 때문에 계속
더 정확한 버전으로 다듬어질 수 있었다.

## 3. 그 밖의 작은 수정들 (P1/P2, 한 줄 요약)

같은 세션에서 독립 코드 리뷰가 찾아낸 나머지 항목들. 코드 자체는 작아서 스터디 노트로
따로 자세히 다루지 않지만, 무엇을 왜 고쳤는지만 기록해둔다.

- **`ChatResponse`에서 `unverifiedReferences` 제거**: 검증 실패한(환각 의심) 구절 주소를
  API 응답에 그대로 노출하고 있었다. 사용자에게 보일 이유가 없는 내부 진단 정보라 서버
  로그로만 남기고 DTO에서 제거.
- **Rate limit 추가**: 사용자별 시간당 요청 횟수 제한이 아예 없었다 — 실수든 악의든
  무제한 API 호출이 가능한 상태였다. 간단한 고정 윈도 카운터로 막았다.
- **Kakao/Google 로그인 예외 구분**: "제공자 서버가 잠깐 안 됨"과 "진짜 위조된 토큰"을
  똑같이 401로 처리하고 있었다. 전자는 사용자 잘못이 아니니 503으로 분리.
- **`BibleTextImporter` 트랜잭션**: 3만 절을 500개씩 배치로 저장하는데 도중에 실패하면
  앞부분은 이미 커밋된 채로 남았다. 전체를 하나의 트랜잭션으로 묶어 실패 시 통째로 롤백.
- **한글 가짜 성경책 이름 우회는 고치려다 되돌림**: "에녹서 3:16"처럼 실재하지 않는
  책 이름이 검증을 우회하는 문제가 있었다. 영어 처리 방식과 똑같이 "해석 실패해도 일단
  보존"하도록 고쳤더니, 기존에 잘 걸러지던 "내일 오후 3:30에 만나요" 같은 **시각 표현**까지
  성경 구절로 오인하는 회귀가 발생했다. 한글은 정규식만으로 "가짜 책 이름"과 "시각·점수
  표현"을 구분할 방법이 없고, 지금까지는 책 이름 해석 성공 여부가 그 구분 필터를 대신하고
  있었던 것이다. 근본적인 해결책(예: 알려진 비-성경 단어 목록, 문맥 기반 판단) 없이는
  못 고치는 문제라 원복하고 "알려진 한계"로 기록만 남겼다 — 겉보기에 간단해 보이는 버그가
  사실은 더 큰 설계 문제였던 사례.
