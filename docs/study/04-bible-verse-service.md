# 04. BibleVerseService — LLM 환각을 DB로 차단하는 설계

## 1. 문제: LLM은 성경 구절을 지어낸다

LLM에게 "위로가 되는 성경 구절 알려줘"라고 하면, 그럴듯한 주소에 그럴듯한 본문을
*생성*한다. 대부분 맞지만 가끔 틀리고, 틀린 한 번이 이 서비스에는 치명적이다.
(존재하지 않는 구절, 비슷하지만 다른 본문, 다른 역본 혼입 등)

**해법 (D-003): 모델과 DB의 역할 분리**

```
모델: "빌립보서 4:6-7이 어울리겠네요"  ← 주소만 제안
서버: DB에서 빌 4:6-7 원문 조회       ← 본문은 여기서만
  ├─ 있으면 → 원문을 응답에 삽입
  └─ 없으면 → VerseNotFoundException = 환각 신호, 거부
```

모델 출력의 구절 *본문*을 사용자에게 직접 보내는 코드 경로는 금지다(헌법).
생성형 AI 서비스의 일반 패턴으로 말하면: **신뢰할 수 없는 생성 결과를
신뢰할 수 있는 저장소로 검증(grounding)** 하는 구조다.

## 2. 구절 주소 파서

입력 형태가 다양하다: `요한복음 3:16`, `요 3:16`, `빌립보서 4:6-7`, `눅 15장 11-32절`.
정규식 named group으로 파싱한다:

```java
"^\\s*(?<book>\\D+?)\\s*(?<chapter>\\d{1,3})\\s*(?::|장)\\s*"
+ "(?<start>\\d{1,3})(?:\\s*[-~]\\s*(?<end>\\d{1,3}))?\\s*절?\\s*$"
```

- `\\D+?` (숫자 아닌 문자, 최소 매칭) — 책 이름엔 숫자가 없다는 도메인 지식을 활용
- `(?::|장)` — `3:16`과 `3장 16절` 두 표기를 모두 수용
- 책 이름 해석은 `BibleBookCatalog`가 담당: 풀네임(`창세기`)과 약어(`창`)를
  공백 제거 후 맵 조회. 66권의 이름·약어는 DB 시드에서 로드한다.

## 3. 존재 검증은 3단계

```java
public VersePassage getPassage(String rawReference) {
    VerseReference ref = parser.parse(rawReference);          // ① 형식 + 책 이름
    if (ref.chapter() > ref.book().getChapterCount())          // ② 장 범위 (DB 조회 전)
        throw new VerseNotFoundException(...);
    List<BibleVerse> rows = repository.findBy...(...);         // ③ DB 원문 조회
    if (rows.size() < ref.verseCount())                        //    범위 일부 누락도 거부
        throw new VerseNotFoundException(...);
    ...
}
```

③의 디테일: `빌 4:6-8`을 요청했는데 6, 7절만 있으면? **행 수와 요청 범위를 비교**해서
하나라도 빠지면 전체를 거부한다. "부분적으로 맞는 응답"이 가장 위험하기 때문.

## 4. 예외를 둘로 나눈 이유

| 예외 | 의미 | 받는 쪽의 해석 |
|------|------|---------------|
| `InvalidVerseReferenceException` | 형식이 틀렸거나 모르는 책 | 입력 자체가 잘못됨 |
| `VerseNotFoundException` | 형식은 맞는데 DB에 없음 | **환각 의심 신호** (QA T8) |

호출부가 "잘못 물어봤다"와 "모델이 지어냈다"를 구분해 다르게 대응할 수 있게 하는 설계.
예외 타입이 곧 도메인 언어가 된다.

## 5. 테스트에서 배운 것

- **픽스처에 진짜 성경 본문을 안 쓴다.** `"(테스트 본문 6절)"` 같은 플레이스홀더만 사용.
  "모델 기억으로 구절 인용 금지" 원칙을 테스트 데이터에도 적용한 것 — 에이전트(나)의
  기억에서 나온 본문 역시 검증 안 된 출처이기 때문이다.
- **단위 테스트 계층 분리:**
  - 파서: 의존성 없는 순수 로직 → new로 직접 생성해 테스트 (가장 빠름)
  - 서비스: Repository만 Mockito mock → DB 없이 분기 로직 검증
  - 리포지토리: `@DataJpaTest` + H2 → 쿼리 메서드와 엔티티 매핑 검증
- 테스트 메서드 이름을 한글로 (`존재하지_않는_장은_DB_조회_전에_거부한다`) — 실패 시
  리포트가 곧 명세서가 된다.

## 6. 임포터 설계 (소스 확정 대기 중)

- `@Profile("bible-import")` — 평소엔 빈 자체가 안 뜨고, 명시적으로 프로파일을 켤 때만 동작.
  "위험한 도구는 평소에 꺼두기" 패턴.
- 이미 본문이 있으면 중단 (중복 적재 가드), 500행 단위 배치 saveAll, 줄 단위 검증
  (책 약어·장 범위·빈 본문) 후 전체 실패 — 부분 적재된 어정쩡한 상태를 만들지 않는다.
