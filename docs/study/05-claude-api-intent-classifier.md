# 05. Claude API 연동과 의도 분류기

## 1. Messages API 한 장 요약

Claude API의 모든 것은 `POST /v1/messages` 하나로 통한다. 공식 Java SDK 기준:

```java
MessageCreateParams params = MessageCreateParams.builder()
        .model("claude-haiku-4-5")   // 어떤 모델로
        .maxTokens(16L)              // 최대 출력 토큰 (분류는 라벨 하나라 16이면 충분)
        .system(systemPrompt)        // 역할·규칙 (시스템 프롬프트)
        .addUserMessage(userMessage) // 사용자 입력
        .build();
Message response = client.messages().create(params);
```

- API는 **무상태(stateless)** — 대화 맥락이 필요하면 매 요청에 이력을 다 보낸다.
- 응답 `content`는 블록 리스트(text/thinking/tool_use 등)라서 `block.text()`로
  텍스트 블록만 걸러 모은다.
- 과금은 입력+출력 토큰량 기준. `maxTokens`를 작게 잡는 건 비용이자 안전장치다.

## 2. 모델 라우팅: 왜 분류는 Haiku인가

| 모델 | 위치 | 이 프로젝트에서의 용도 |
|------|------|----------------------|
| Haiku | 가장 싸고 빠름 | **의도 분류** (라벨 하나 출력), 일상 대화(예정) |
| Sonnet | 중간 | 신앙 상담·지식QA 응답(예정) |
| Opus | 가장 강력 | (현재 계획 없음 — 비용) |

분류는 "6지선다"라 최고 지능이 필요 없다. PRD의 단위 경제성(사용자당 월 300~800원으로
구독 3,900원에서 마진 확보)은 **요청마다 적절한 모델을 고르는 라우팅**이 전제다 (D-010).
모든 요청에 최상위 모델을 쓰면 경제성이 무너진다. "모델 선택도 아키텍처 결정"이라는 것.

## 3. SDK를 인터페이스 뒤에 숨기기

```java
public interface ClaudeChat {
    String complete(String model, int maxTokens, String systemPrompt, String userMessage);
}
```

`IntentClassifier`가 SDK 타입(`AnthropicClient`)이 아니라 이 인터페이스에 의존하게 했다. 이유:

1. **테스트** — Mockito로 `ClaudeChat`만 모킹하면 실제 API 호출(비용·네트워크·키) 없이
   분류 로직을 단위 테스트할 수 있다.
2. **교체 용이** — SDK 버전업, 스트리밍 전환, 재시도 로직 추가가 구현체 한 곳에 격리된다.

전형적인 의존 역전(DIP). "외부 서비스는 포트 뒤에 둔다"는 헥사고날 아키텍처의 축소판.

## 4. 분류 프롬프트 설계

프롬프트는 자바 문자열이 아니라 **리소스 파일**(`prompts/intent-classifier.txt`)로 뒀다.
프롬프트 변경은 사람 승인 대상이라, diff가 깨끗하게 보이는 별도 파일이 검토에 유리하다.

설계 포인트:
- 6개 범주(위기/상담/기도문/지식QA/일상대화/범위밖)의 **정의와 예시**를 명시
- 규칙 1: "위기 신호가 조금이라도 의심되면 무조건 위기" — 안전 우선 타이브레이커
- 규칙 2: "범주 이름 하나만 출력" — 파싱 가능한 출력 강제

## 5. 신학 검사가 잡아낸 치명적 결함 (이 장의 핵심)

독립 검사 에이전트(theology-checker)에게 이 프롬프트를 검토시켰더니 **FAIL(critical)**:

> 모델이 위기를 올바르게 감지하고도 출력 형식을 어기면(예: `"위기 (자살 암시)"`)
> 정확 일치 파싱에 실패 → 폴백 "상담"으로 **강등**된다. 키워드 필터가 놓친 간접 표현을
> 잡는 게 이 분류기의 존재 이유인데, 정확히 그 지점에 구멍이 있다.

수정 (D-011):

```java
// 출력에 "위기"가 포함되면 형식과 무관하게 CRISIS
if (label != null && label.contains(Intent.CRISIS.label())) {
    return Intent.CRISIS;
}
```

여기서 배울 일반 원칙 — **LLM 출력 파싱의 안전 설계**:

1. LLM은 형식 지시를 *대체로* 따르지만 *항상* 따르지는 않는다. 파서는 형식 위반을 전제해야 한다.
2. 파싱 실패의 폴백이 안전 critical 분기를 우회하면 안 된다. "엄격한 파싱 + 안일한 폴백"
   조합이 가장 위험하다.
3. 안전 critical 라벨은 관대하게(부분 일치), 나머지는 엄격하게 — 비대칭 파싱이 정답일 수 있다.

또한 검사가 제안한 **프롬프트 본문 수정 3건**(제3자 위기 신호, 혼합 예시, 기도문 경계 문구)은
위기 프로토콜 관련이라 즉시 반영하지 않고 사람 승인 대기로 남겼다.
"코드 수정(내 권한) vs 프롬프트 수정(사람 승인)"을 가르는 운영 규율이 실제로 작동한 사례.

## 6. API 키 관리

- 키는 `ANTHROPIC_API_KEY` 환경변수로만 주입. 코드·설정 파일에 키를 넣지 않는다.
- 키가 없어도 **부팅은 되게** 했다 (경고 로그 + 호출 시점 실패). DB 작업 등 키가 필요 없는
  로컬 개발을 막지 않기 위한 트레이드오프. 반대로 "키 없으면 부팅 실패"가 맞는 서비스도 있다
  (API가 핵심 기능의 전부일 때). 어느 쪽이든 *의식적으로* 골라야 한다.
