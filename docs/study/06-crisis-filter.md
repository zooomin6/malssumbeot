# 06. CrisisFilter — 위기 감지 파이프라인

## 1. 왜 "결정론적 필터"가 LLM보다 앞에 있나

위기 감지를 LLM에만 맡기면 안 되는 이유:

- LLM 호출은 **실패할 수 있다** (네트워크, 키, 장애). 실패 순간에 위기 감지가 같이 죽으면 안 된다.
- LLM은 **확률적**이다. 같은 입력에 늘 같은 판정을 보장하지 않는다.
- LLM 호출은 돈과 시간이 든다. 정규식은 공짜고 마이크로초 단위다.

그래서 방어선을 이중으로 쌓는다 (defense in depth):

```
사용자 메시지
   │
   ▼
[1차] CrisisFilter   ← 결정론적 정규식. 빠르고, 절대 안 죽고, 항상 같은 판정
   │ (통과 시)
   ▼
[2차] IntentClassifier ← LLM(Haiku). 키워드가 못 잡는 맥락적·간접적 표현 감지
   │
   ▼
[3차] 응답 프롬프트 내 위기 프로토콜 (예정) ← 모든 모드에 내장
```

각 층은 서로 다른 *종류*의 실수를 보완한다: 1차는 변형 표현에 약하고("힘들어서 그만하고
싶다"), 2차는 형식 위반·장애에 약하다(05장의 critical 결함). 셋이 겹쳐야 안전망이 된다.

## 2. 패턴 매칭의 디테일

```java
String normalized = message.replaceAll("\\s+", "");  // "죽 고 싶 어" → "죽고싶어"
```

- **공백 정규화 후 부분 매칭** — 띄어쓰기 변형으로 필터를 피하는 것을 막는다.
- 패턴은 코드가 아니라 **리소스 파일**(`crisis/crisis-patterns.txt`)에 둔다.
  위기 감지 기준은 사람이 검토해야 하는 영역이라, diff가 잘 보이는 데이터 파일이 적합하다.
- 패턴 파일이 비어 있으면 **부팅 자체를 실패**시킨다 (fail-fast). 빈 패턴으로 조용히
  돌아가는 것 = 위기 감지가 무력화된 채 서비스되는 것. 가장 나쁜 실패는 조용한 실패다.

## 3. 오탐 vs 미탐 — 비대칭 트레이드오프

분류기에는 두 가지 오류가 있다:

| 오류 | 이 서비스에서의 결과 | 비용 |
|------|---------------------|------|
| 오탐 (false positive) | 평범한 사용자에게 위기 안내가 나감 | 어색함 — 회복 가능 |
| 미탐 (false negative) | 위기 사용자가 일반 응답을 받음 | **돌이킬 수 없음** |

비용이 비대칭이면 임계값도 비대칭이어야 한다. 그래서 "애매하면 위기"(재현율 우선)가
원칙이고, `때리` 같은 다소 공격적인 패턴도 포함했다. 이건 ML의 precision/recall
트레이드오프를 규칙 기반 시스템에 적용한 것과 같다.

## 4. Sticky 상태 — 멀티턴의 함정

theology-checker가 지적한 시나리오:

```
턴 1: "요즘 다 끝내고 싶어"        → 위기 감지 ✓
턴 2: "내일 면접 기도문 써줘"      → 평범한 메시지... 위기 컨텍스트 휘발?
```

메시지 단위로만 판단하면 턴 2에서 위기 상태가 사라진다. 해결: **세션 단위 sticky flag** —
한 번 위기가 감지되면 유지 시간(기본 30분, 설정값) 동안 위기 프로토콜을 계속 적용한다.

```java
public CrisisCheck check(String sessionId, String message) {
    if (detector.detect(message).isPresent()) {       // 이번 메시지에서 감지
        sessionStore.mark(sessionId);                  // 세션에 도장
        return CrisisCheck.newSignal(...);
    }
    if (sessionStore.isActive(sessionId)) {            // 직전 감지의 여운
        return CrisisCheck.sticky();
    }
    return CrisisCheck.none();
}
```

구현 디테일:
- `ConcurrentHashMap<String, Instant>` 인메모리 — 지금은 단일 서버 전제.
  REST API + 세션이 생기면 Redis/DB로 옮긴다 (서버 여러 대면 인메모리는 공유 안 됨).
- 만료는 읽기 시점에 게으르게(lazy) 처리 — 별도 청소 스레드 없이 단순하게.
- 재감지 시 유지 시간 갱신 (위기 대화가 이어지는 동안 계속 연장).

## 5. 시간을 테스트하는 법: Clock 주입

"30분 뒤 만료"를 테스트하려고 실제로 30분 기다릴 수는 없다. `java.time.Clock`을
주입받게 하고, 테스트에서는 시간을 마음대로 돌리는 가짜 시계를 쓴다:

```java
clock.advanceSeconds(31 * 60);                  // 31분 점프
assertThat(store.isActive("session-1")).isFalse();
```

`Instant.now()`를 코드에 직접 박으면 이 테스트는 불가능하다. **"현재 시각"도 의존성**이고,
의존성은 주입한다 — 시간 관련 로직이 있는 모든 코드에 적용되는 패턴.

## 6. 테스트가 운영 패턴 파일을 직접 쓰는 이유

탐지 테스트는 별도 픽스처가 아니라 **운영과 동일한 `crisis-patterns.txt`** 를 로드한다.
픽스처를 따로 만들면 "테스트는 통과하는데 운영 패턴에는 구멍"인 상태가 가능해진다.
이 테스트들은 사실상 패턴 파일의 회귀 테스트다: 누군가 패턴을 잘못 지우면
`직접_표현을_감지한다`가 깨진다.

## 7. 남은 일

  검사를 건너뛸 수 없게 (헌법: "위기 감지는 의도 분류보다 앞단, 우회 불가").
