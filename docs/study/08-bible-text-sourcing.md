# 08. 개역한글 본문 확보 — 저작권 조사와 스크래핑

## 1. "무료니까 써도 된다"는 착각

개역한글은 저작권이 만료된 퍼블릭 도메인(D-001)이지만, 이번 세션에서 처음 후보로 나온
`holybible.or.kr`은 실제로 못 쓰는 곳이었다. 페이지 하단 고지를 직접 읽어보니:

> "당 사이트에서 사용하는 컨텐츠에 대한 저작권을 가진 기관의 요청에 따라
> **모바일앱에서 당 사이트로의 어떤 형태의 연동/연결도 금지** 합니다."

핵심 교훈: **"본문 자체가 퍼블릭 도메인인가"와 "이 사이트에서 긁어와도 되는가"는 별개 질문이다.**
같은 텍스트라도 어느 사이트에 올라와 있느냐에 따라 그 사이트 자체의 이용약관이 덧씌워질 수
있다. 그래서 후보 3곳(홀리바이블, 위키문헌, 대한성서공회 자체 페이지)을 각각 이용약관까지
확인했고, 최종적으로 **저작권자 본인(대한성서공회)의 공식 페이지**를 골랐다 — 판본·정확성
리스크가 가장 낮고, 애초에 저작권자 본인 사이트라 별도 약관 충돌 여지가 없기 때문이다(D-016).

## 2. 재산권은 만료돼도 인격권은 안 만료된다

대한성서공회 공식 FAQ를 직접 읽고 확인한 부분:

- **저작재산권**: 개역한글(1961.7.10 발행) → 2011.12.31 만료. 로열티 없이 자유 사용 가능.
- **저작인격권**(성명표시권·동일성유지권): **영구 존속**. 만료되지 않는다.

그래서 "재산권 만료 = 아무렇게나 써도 됨"이 아니다. 본문을 임의로 고치면 안 되고(우리는
DB 원문 그대로만 쓰므로 구조적으로 이미 지켜지고 있음, D-003), 어딘가에 "성경전서
개역한글판, 대한성서공회"라고 출처 표시를 해야 한다 — 이건 RN 앱 작업 때 넣을 새 할 일로
PROGRESS.md에 남겨뒀다.

## 3. Java로 바로 안 짜고 파이썬으로 먼저 검증한 이유

HTML 파싱 로직(각주 제거, 절 경계 판단)을 처음부터 Java/Jsoup으로 작성하지 않고, 먼저
파이썬 BeautifulSoup으로 실제 사이트 응답을 놓고 프로토타입했다:

```python
for tag in root.select('a.comment, div.D2, font.smallTitle, font.chapNum'):
    tag.decompose()
```

이유는 단순하다 — **반복 실행 비용이 다르다.** 파이썬 스크립트는 한 줄 고쳐서 바로
재실행하며 결과를 눈으로 확인할 수 있지만, Java는 컴파일이 껴서 사이클이 느리다. 정답이
뭔지 모르는 상태(사이트 HTML 구조를 처음 보는 상태)에서는 빠르게 틀려보는 게 중요하므로,
가벼운 언어로 먼저 답을 찾고, 답이 확정된 다음에 실제 프로덕션 언어(Java)로 그대로
옮겼다. 두 구현이 같은 결과를 내는지 실제 데이터(시편 3편·창세기 1장·요한복음 3장)로
대조 검증한 뒤에야 Java 코드를 "완성됐다"고 판단했다.

## 4. 정규식 대신 DOM 구조를 믿은 이유

원래는 "절 번호는 숫자로 시작한다"는 정규식 패턴으로 절을 나누려 했다. 그런데 실제 HTML을
까보니 사이트가 이미 의미 있는 라벨(class)을 붙여놓고 있었다:

```html
<span class="number">2&nbsp;&nbsp;&nbsp;</span>본문...
<a class="comment" onClick="clickPopUp('D_172020_1', event)">1)</a>   ← 각주 마커
<div class="D2" style="display:none">구원</div>                       ← 각주 내용(숨김)
```

정규식으로 "숫자면 절이다"라고 추측하면, 각주 마커("1)")도 숫자로 시작해서 절 번호와
헷갈릴 여지가 있다. 반면 `class="number"` vs `class="comment"`는 사이트가 이미 구분해둔
사실이라 추측이 필요 없다. **원본에 구조(라벨)가 있으면 그걸 믿고, 없을 때만 패턴 추론으로
간다** — 이게 스크래핑 코드를 안정적으로 짜는 원칙이다.

## 5. "18-19" — 처음엔 없던 케이스

전체 66권을 다 돌리다가 신명기 어딘가에서 절 번호가 "18"이 아니라 "18-19"로 나오는
경우를 만나 `NumberFormatException`으로 죽었다. 두 절을 한 문장으로 묶어 번역한 구간을
표기하는 방식이었다. 여기서 판단이 필요했다:

- 18절에만 넣고 19절을 비우면? → 나중에 `BibleVerseService`가 "신명기 X:19"를 실제로
  존재하는 절인데도 "DB에 없음"으로 판정해 환각(hallucination)으로 오분류한다. 절대
  원칙 2·T8과 정면으로 충돌.
- 그래서 **18절과 19절 둘 다에 동일한 본문을 넣는 쪽을 택했다.** 인쇄상으로는 합쳐진
  표현이지만, 사용자가 어느 절 번호로 인용을 요청하든 실제 존재하는 절로 인식되게 하는 게
  우선이다.

이런 예외는 미리 다 예측할 수 없다. 그래서 **1,189장 전체를 돌리기 전에 "책 완료" 로그를
찍게 만들어서**, 실패하면 몇 권째에서 멈췄는지 바로 알 수 있게 했다 — 40분짜리 배치
작업에서 중간 실패를 빨리 눈치채는 것도 설계의 일부다.

## 6. 검증은 "숫자가 맞다"만으로는 부족하다

적재 후 확인한 3단계:

1. **총 절 수**: 31,102 (스크래핑 결과와 DB 저장 결과가 정확히 일치 — 데이터 손실 없음)
2. **책별 최대 장 번호**: `bible_book.chapter_count`와 실제 저장된 최대 장이 66권 전부 일치
   (0건 불일치 — SQL `HAVING` 절로 한 방에 확인)
3. **육안 확인**: 창세기 1:1, 시편 23:1, 요나 1:1, 요한복음 3:16 — 잘 아는 구절을 직접
   눈으로 읽어서 각주 잔여물이나 잘림이 없는지 확인

1·2번은 "숫자가 맞다"는 것만 보여준다. 절이 다 있어도 본문 내용이 깨져 있으면 숫자
검증은 못 잡는다. 그래서 **사람이 읽어서 아는 문장인지 확인하는 단계**가 마지막에 꼭
필요했다 — 특히 신앙 서비스처럼 "구절 내용의 정확성"이 곧 서비스 신뢰도인 경우엔 더더욱.
